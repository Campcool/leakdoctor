# 灰汰郎 LINE Bot 後台規劃書

狀態：規劃完成，待業主確認 §14 後開工
建立：2026-07-10｜對應待辦見 AI-README.md §7

## 1. 目標

把「詢價 → 接單 → 派工 → 敲日期 → 完工 → 收款對帳 → 結案」整條流程搬進 LINE，由 AI 自動處理 8 成對話與流轉，人只介入關鍵決策。三種角色都在 LINE 上完成工作：

- **客戶**：問問題、得到報價、下訂、收到提醒
- **夥伴（師傅）**：收派單、接單、與客戶敲日期、回報完工或無法施作
- **老闆**：即時通知、一鍵核准、對帳確認、看日報

## 2. 系統架構

```
客戶/夥伴/老闆 LINE ──webhook──▶ Cloudflare Workers（單一後端）
                                    ├─ 路由：依 userId 判斷角色（客戶/夥伴/老闆）
                                    ├─ AI 對話引擎：Claude API（tool use）
                                    │    └─ 知識庫：價目表+FAQ（從 llms.txt 同步）
                                    ├─ 狀態機：訂單生命週期（§3）
                                    ├─ D1 資料庫（SQLite）：orders/customers/partners/events/payments
                                    ├─ Cron Triggers：提醒、逾時追蹤、日報
                                    └─ 管理頁：Workers 上的簡易後台（老闆用，Basic Auth）
```

**技術選型與理由**
| 項目 | 選擇 | 理由 |
|---|---|---|
| 後端 | Cloudflare Workers + D1 | 免伺服器維運、免費額度大（10萬請求/日）、內建 cron、跟靜態站互不干擾 |
| Bot 通道 | 現有 LINE OA（@478xvlgl）啟用 Messaging API | 客戶已加的好友直接沿用，不用換帳號 |
| AI | Claude API（Haiku 為主，複雜對話升級 Sonnet） | tool use 做結構化動作（建草稿/查價/轉人工），成本低 |
| 程式庫 | **新開 repo**（建議 `Campcool/huitailang-bot`） | 後端含密鑰與邏輯，不可放在 GitHub Pages 公開站的 repo |

**重要限制**：啟用 Messaging API 後，「自動回應訊息」與 AI 回覆並存需在 LINE 後台把回應模式設為 Webhook。一對一手動聊天（聊天模式）與 bot 可切換，設計上保留「轉人工」開關。

## 3. 訂單狀態機（核心）

```
[詢價中] --AI收齊資料--> [草稿] --客戶確認--> [待派單]
   |                                        |
   |（網站表單進線直接建草稿）                --自動/老闆指定夥伴--> [已派單]
                                            |
              +--- 夥伴逾時未回(4hr) --- 提醒/改派 ---+
              |                                        |
        [夥伴已接] --夥伴↔客戶敲日期(bot居中轉達)--> [已預約]
              |                                        |
        夥伴按「無法施作」                      --D-1自動提醒雙方-->
              |                                        |
        [車馬費結案]                            [施工日] --夥伴回報-->
                                                   ├─ [完工待收款] --對帳(§7)--> [已收款] --> [結案]
                                                   └─ [無法施作] --> [車馬費結案]
任一階段：客戶取消 → [已取消]（記錄原因）；老闆可隨時手動改狀態
```

每次狀態變更寫入 `events` 表（誰、何時、從哪到哪、附註）＝完整稽核軌跡，也是日報與對帳依據。

## 4. 客戶端對話設計

1. **網站表單進線**（現成整合點）：網站表單送出的訊息格式固定為「【灰汰郎 到府服務詢價】姓名/電話/地址/項目/日期/時段」，bot 用規則解析（不需 AI）直接建草稿，回覆確認卡＋預估價。
2. **自由對話進線**：AI 負責
   - FAQ 問答（知識庫＝價目表＋服務範圍＋現有文章 FAQ）
   - Slot filling：缺什麼問什麼（服務→機型/坪數→地區→日期偏好→姓名電話地址）
   - 即時估價：規則計算（同網站估價器邏輯），AI 不得自行編價格
3. **確認下訂**：Flex 訊息卡片列出草稿內容＋預估價＋「確認預約／修改／找真人」按鈕。
4. **護欄**：
   - 價格、優惠、保固承諾一律出自規則表，AI 只轉述
   - 任何時候輸入「真人」或 AI 連續 2 次無法理解 → 標記轉人工＋push 通知老闆
   - AI 不處理：殺價、客訴、賠償——直接轉人工

## 5. 夥伴端流程

- **註冊**：夥伴加同一 OA，老闆在管理頁把該 userId 標為夥伴（設服務類型＋地區）
- **派單**：新訂單 push 派單卡（服務/地區/概況/預估價，隱去客戶完整電話）→「接單／無法接」
- **派單規則**：v1 = 老闆手動指定或群發搶單（先接先得）；v2 = 依地區+服務類型自動匹配
- **敲日期**：夥伴接單後，bot 居中：夥伴提 2-3 個時段 → bot 轉給客戶選 → 選定即鎖日期進[已預約]，雙方都收到確認卡。（不直接交換聯絡方式，保護平台；施工前一日才給夥伴客戶電話）
- **完工回報**：施工日 bot 主動問夥伴 →「已完工（可附照片）／無法施作」
  - 已完工 → 進入收款對帳
  - 無法施作 → 選原因（現場評估後客戶不做/條件不符/其他）→ 進[車馬費結案]，bot 通知客戶車馬費金額與付款方式

## 6. 老闆端

- **即時 push**：新草稿、客戶確認下訂、夥伴接單/拒單、完工回報、轉人工、車馬費結案
- **一鍵操作**：通知卡上直接按「核准派單／指定夥伴／改價／取消」
- **指令**：輸入「今日」「本週」「未結案」查清單；「訂單 #123」看詳情
- **每日 21:00 日報**：新詢價 X、成交 X、完工 X、待對帳 X 元、逾時未處理清單
- **管理頁**（網頁）：訂單列表/篩選/匯出 CSV、夥伴管理、價目與車馬費規則表編輯

## 7. 對帳設計（三階段，逐步升級）

| 階段 | 做法 | 自動化程度 |
|---|---|---|
| v1 手動確認 | 完工後 bot 發付款資訊（轉帳帳號）給客戶；客戶回報「已轉帳＋末五碼」；老闆對過存摺後在通知卡按「確認收款」→ 自動結案 | 半自動（人看帳） |
| v2 末五碼比對 | 老闆每天把網銀交易明細 CSV 丟到管理頁，系統比對金額+末五碼自動銷帳，對不上的列異常清單 | 8 成自動 |
| v3 金流全自動 | 接綠界/藍新付款連結（信用卡/ATM虛擬帳號），webhook 進帳即自動銷帳結案 | 全自動（有手續費 ~2%+，且需申請商店） |

建議 v1 上線 → 量起來後直上 v3。**銀行帳戶 API 個人戶拿不到，純轉帳無法真全自動，這是台灣現實限制。**

## 8. 資料模型（D1）

```
customers(id, line_user_id, name, phone, address, created_at, note, blacklist)
partners(id, line_user_id, name, phone, services[], areas[], active)
orders(id, customer_id, partner_id, service, detail_json, area, address,
       est_price, final_price, travel_fee, status, scheduled_at,
       created_at, closed_at, source[web_form|chat], cancel_reason)
events(id, order_id, actor, from_status, to_status, note, created_at)
payments(id, order_id, amount, method, last5, confirmed_by, confirmed_at)
conversations(line_user_id, role, last_intent, handoff_flag, updated_at)
```

## 9. 排程（Cron Triggers）

- 每小時：夥伴接單逾時 4hr 提醒、8hr 通知老闆改派；客戶草稿 24hr 未確認發一次跟進
- 每日 09:00：今日施工清單推老闆＋各夥伴
- 每日 18:00：明日施工 D-1 提醒客戶＋夥伴
- 每日 21:00：老闆日報
- 完工後 +2 日：發客戶滿意度／好評邀請（導 Google 商家評論——商家檔案建好後）
- 完工後 +11 個月（冷氣/洗衣機）：回購提醒（年度清洗到期）

## 10. 業主沒提到、但規劃內建的補強

1. **評價收集**：完工自動邀評，好評導 Google 商家（信任資產從零開始累積）
2. **回購行銷**：清洗類服務天然年週期，DB 有日期就能自動喚回（免費營收）
3. **奧客/黑名單標記**：取消 2 次以上、爽約自動標記，派單卡上警示夥伴
4. **個資**：地址電話僅存必要欄位、結案 1 年後遮罩；夥伴只在施工前一日拿到完整電話
5. **車馬費規則表**：依地區設定（管理頁可編），報價卡自動帶入
6. **月報**：每月 1 號自動彙總上月營收/成交率/服務占比/各夥伴業績
7. **LINE 訊息額度監控**：push 用量接近方案上限時提醒老闆升級

## 13. AI 能做到什麼程度（誠實評估）

**AI（我或接手的 AI）可以直接完成 ≈90%：**
- 全部程式碼：Workers 後端、狀態機、Flex 卡片、AI 對話引擎、D1 schema、管理頁、cron
- 本地/預發環境測試（模擬 webhook 事件跑完整個狀態機）
- 部署腳本與文件、AI-README 同步

**必須業主親手做（AI 無法代辦）：**
1. LINE Developers Console：把 OA 啟用 Messaging API，取得 Channel Secret / Access Token，回應模式改 Webhook（約 15 分鐘，AI 給圖文步驟）
2. Cloudflare 帳號註冊＋授權部署（或給 API Token 讓 AI 部署）
3. Anthropic API key 申請（AI 對話引擎的鑰匙）
4. 若做 v3 對帳：向綠界/藍新申請商店（需公司/行號或個人賣家資料）
5. 上線前用自己手機實測一輪、決定價目與車馬費規則表的最終數字

**AI 做不到/不該做：** 殺價與客訴判斷（設計為轉人工）、銀行帳戶直連（台灣個人戶無此 API）、代表業主簽金流合約。
